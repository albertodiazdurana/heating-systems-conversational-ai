## Title

Plan documents in `dsm-docs/plans/` may silently violate DSM Template 8
(no enforcement at creation time, no audit at session start)

## Discovery context

Mid-session, while preparing to close Sprint 1 in this project, the user
asked: "Why does the plan not include a description for Sprint boundary
checklist? The structure of the plan is not complete. Find the root cause."

Investigation traced through three layers and found a real structural gap.

## Three-layer root cause

### Layer 1: Plan author skipped Template 8

Sprint 1 plan
([`dsm-docs/plans/2026-04-07_sprint1_langgraph_plan.md`](https://github.com/albertodiazdurana/heating-systems-conversational-ai))
has 9 sections: Scope (1), Stack (2), File layout (3), Build order (4),
Design decisions (5), Exit criteria (6), Deferred (7), Open questions (8),
Gates summary (9).

DSM_2.0.C §1 Template 8 mandates a different and richer structure:

| Mandated section (Template 8) | Present in Sprint 1 plan? |
|---|---|
| Header (Duration, Goal, Prerequisites) | Partial (no Duration, no Prerequisites) |
| Research Assessment | No |
| Experiment Gate | No |
| Branch Strategy | No |
| Deliverables (MUST / SHOULD / COULD) | Yes (in §1 Scope, but not labeled this way) |
| Phases (with execution mode + DSM references + success criteria) | No (replaced by ad-hoc §4 Build order) |
| **Phase Boundary Checklist (intra-sprint)** | **No** |
| Open Design Questions | Yes (§8, slightly different label) |
| How to Resume | No |
| **Sprint Boundary Checklist** | **No** |

Whoever drafted the plan either did not know about Template 8 or chose to
ignore it. Result: a non-compliant plan sits in `dsm-docs/plans/` and drives
sprint execution toward an undocumented closure.

### Layer 2: No enforcement at plan creation time

DSM has `/dsm-align` which validates folder structure and CLAUDE.md
alignment. It does **not** validate that plan documents in `dsm-docs/plans/`
follow Template 8. There is no `/dsm-plan-align` or equivalent skill.

A non-compliant plan can sit in `dsm-docs/plans/` indefinitely without any
tool flagging it. Sprint 2 and Sprint 3 plans (when written) will inherit
the same pattern unless we fix the source-of-truth or add enforcement.

### Layer 3: Session-start gate audits artifacts, not plan structure

[`/dsm-go` Step 3.6 "Sprint boundary gate"](file:///home/berto/dsm-agentic-ai-data-science-methodology/scripts/commands/dsm-go.md)
verifies that boundary **artifacts exist** (checkpoint, blog entry,
feedback files) when MEMORY.md says a sprint completed. It does **not**
verify that the plan document itself contains a Sprint Boundary Checklist
section.

So even when the gate fires, it audits filesystem artifacts, not the
plan-document structure that is supposed to *drive* those artifacts. A
plan can omit the Sprint Boundary Checklist entirely, and the session-
start gate will not catch it.

## Concrete consequences observed in this session

- We were about to close Sprint 1 without a written Sprint Boundary
  Checklist in the plan to drive the closure. The checklist exists
  conceptually (we'd be doing checkpoint + blog + feedback + README anyway)
  but is not anchored in the plan, so the closure depends on the agent's
  memory of what `/dsm-wrap-up` happens to do, not on the plan.
- The Phase Boundary Checklist (intra-sprint) is also missing, which is
  why Sprint 1 had ad-hoc decision-tracking ("BL-002 edit 4") instead of
  structured per-phase observations that would feed `methodology.md`.
- Sister plan in this repo
  (`dsm-docs/plans/2026-04-07_e2e_hybrid_backbone.md`) has a different
  10-section structure that overlaps with Template 8 partially. Audit of
  other spokes likely surfaces the same pattern.

## Backlog proposals (three, ordered by leverage)

### BL-A: Extend `/dsm-align` with a plan-document audit step

Add a new step to `/dsm-align` (between current Step 5 and Step 6) that
scans every `.md` file in `dsm-docs/plans/` (excluding `README.md` and
files in `done/`) and validates structural compliance with DSM_2.0.C
Template 8.

Validation criteria:
- Required top-level sections present (header, deliverables, phases,
  exit criteria, sprint boundary checklist, etc.)
- Sprint Boundary Checklist contains the 8 mandatory bullets per
  Template 8 §1 lines 144-151
- Phase Boundary Checklist present if plan has multi-phase structure

Behavior: report non-compliance as warnings (not auto-fix), include
specific missing-section names per file. Do NOT auto-add sections —
plan content is project-specific judgment, only structure is validated.

**Pros:** centralizes plan validation in the existing alignment skill;
zero new commands; reuses existing reporting infrastructure.
**Cons:** `/dsm-align` is already a long skill; adds another step.

### BL-B: Ship a dedicated `/dsm-plan-align` (or `/dsm-plan-init`) skill

New skill that handles plan-document creation and validation as a
first-class concern:
- `/dsm-plan-init <sprint-name>` scaffolds a new plan from Template 8
- `/dsm-plan-align` validates existing plans (same checks as BL-A but
  invocable independently)

**Pros:** clean separation of concerns; new sprints get a compliant
skeleton automatically.
**Cons:** another command; users must remember to invoke it.

### BL-C: Add a session-start hard gate against non-compliant plans

Modify `/dsm-go` Step 3.6 "Sprint boundary gate" to also verify that the
plan document for the active sprint contains a Sprint Boundary Checklist
section before suggesting any sprint-close work. If missing, prompt: "The
plan for Sprint N does not contain a Sprint Boundary Checklist. Add one
before closing the sprint."

**Pros:** catches the gap exactly when it matters (just before closure);
no new skill needed.
**Cons:** fires only at sprint closure; non-compliant plans drift through
the sprint until the user tries to close it.

### Recommended combination

BL-A (extend /dsm-align) + BL-C (hard gate at /dsm-go Step 3.6). Together
they catch non-compliance at every alignment run AND at sprint closure,
without needing a separate skill. BL-B can be deferred unless plan
creation friction is a separate problem.
