## BL proposal: /dsm-align Step 12 report should emit explicit `N/A (not DSM Central)` for `Command sync` on spoke runs

### Problem

On spoke projects, `/dsm-align` Step 11 (Check command file drift) is skipped by the first-bullet guard ("Skip this step if the project is not DSM Central"). Step 12's report template lists `Command sync:` as a required field with possible values `OK: N | Drifted: N | Missing: N, or "N/A (not DSM Central)"`. The value is enumerated but the skill does not prescribe *when* the N/A branch applies, and Step 11 itself does not signal Step 12 about the skip.

Result: the field is left to the agent's discretion. In Session 10 of this project, the agent ran an out-of-scope `diff -q` on user-scope runtime command files (`~/.claude/commands/dsm-go.md`, `dsm-wrap-up.md`), found drift, and populated the `Command sync` field with `Drifted: 2` plus a spoke-action recommendation. The finding was out-of-scope for a spoke `/dsm-align` run (user-scope concern, not spoke-scope). Cost: ~115 transcript lines of correction.

### Proposed change

In `scripts/commands/dsm-align.md` Step 12, change the `Command sync` report line specification from enumerated values to a conditional default:

> `Command sync: N/A (not DSM Central)` — when Step 11 was skipped.
> `Command sync: OK: N | Drifted: N | Missing: N` — when Step 11 ran.

This is a one-line wording change in the report template. It closes the "blank field invites invention" failure mode observed in S10 by making the N/A output mandatory and automatic on spokes.

### Priority

Medium. The current behavior does not break /dsm-align's functional correctness (the skill did the right thing; only the report template left a door open for agent-side invention). But it is a recurring risk surface: every spoke run allows the same failure mode.

### Optional companion BL: separate `dsm-check-runtime-commands` skill

If the recurring concern is "is my `~/.claude/commands/` in sync with Central?", that is a user-scope, once-per-machine-per-version check. It does not belong in any spoke-scoped skill. A separate skill run from Central (or a Central wrap-up step) would house it cleanly. Lower priority than the Step 12 wording change; proposed only if the ecosystem wants to formalize user-scope drift as a first-class concern.

## BL proposal: Skill scope is authoritative — explicit principle in DSM_0.2 §8.6 or adjacent

### Problem

DSM_0.2 §8.6 Skill Self-Reference Protocol prevents *memory-based* claims about skill behavior by requiring the agent to read the skill file before answering "does skill X do Y?". It does not prevent *supplementation*: the agent invokes the skill correctly, but then runs additional checks adjacent to the skill's scope and folds their findings into the skill's report. The end-state symptom is the same (the skill appears to say something it didn't), but the mechanism is different.

Observed in S10: agent ran a correctly-invoked `/dsm-align`, then added a `diff -q` on runtime command files outside Step 11's scope, and reported the finding under the `Command sync` field as if the skill had produced it.

### Proposed change

Add a principle to DSM_0.2 §8.6 (or a sibling subsection under DSM_0.2 §8):

> **Skill scope is authoritative.** A skill's declared scope defines its output. Checks adjacent to the skill's scope are separate findings and must be reported separately, not folded into the skill's report. **Silence from the skill on a concern is the skill's answer.** Before widening a skill's report with an adjacent check, ask: is this check part of the skill's documented steps? If not, either (a) run it as an out-of-band audit with its own label, or (b) file a BL to extend the skill.

### Priority

Medium. Prevents a class of errors that does not surface until the user notices the out-of-band framing. Not urgent, but the cost of a single instance (S10 case: ~115 transcript lines) justifies codification.

Session 10 (2026-04-23) methodology observation documented in `2026-04-23_s10_methodology.md` (same feedback batch).
