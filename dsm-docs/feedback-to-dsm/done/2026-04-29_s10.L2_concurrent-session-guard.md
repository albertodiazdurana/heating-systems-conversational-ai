**Project:** heating-systems-conversational-ai

**Date:** 2026-04-29
**DSM version at time of observation:** v1.8.0

## What happened

Two Claude Code conversations were running in parallel against the same project (`heating-systems-conversational-ai`) on the same git branch (`session-10/2026-04-23`). Neither agent detected the other. Discovery happened only when `git status` and the shared `.claude/session-transcript.md` showed entries the active agent had not authored:

- This (S10.L2) session ran `/dsm-align` v1.7.0 → v1.8.0, drafted the Haystack upstream issue (T6), and closed BL-004 (Sprint 1 provenance audit).
- A parallel session ran T7 (Phase 2 embedding-model selection), wrote `dsm-docs/decisions/2026-04-24_phase2-embedding-model-selection.md`, and made 3 edits to `dsm-docs/plans/2026-04-07_e2e_hybrid_backbone.md`.

Both sessions were on the same branch, modifying the same working tree, appending to the same transcript file, with no awareness of each other.

---

## Methodology observation , two failure layers

### Layer 1 , no concurrent-session detection

DSM_0.2 has no protocol that detects or prevents a second concurrent session on the same project. `/dsm-go` Step 5.8 detects *prior incomplete wrap-ups* (branch session number > MEMORY session number) but has no concept of a *currently running* session. Result: two windows can operate the same branch in parallel without either knowing.

Rule that should hold: **every session must be wrapped, light or full.** A session that has not been wrapped is, by definition, still active. Starting a second session without confirming the first is wrapped breaks the single-session-per-project assumption that every other DSM protocol depends on (transcript continuity, baseline file, MEMORY.md latest-session pointer, branch state).

### Layer 2 , auto mode suppressed `/dsm-go` Step 5.9 prompt

`/dsm-go` Step 5.9 already has the right prompt for the light-wrap-continuation case:

> *If `type: light`, the previous session ended with a light wrap-up, signaling continuation. Prompt the user: "Last session ended with a light wrap-up (continuation expected). Switch to /dsm-light-go for a faster resume? (y = switch to light-go, n = continue with full go)".*

The S10.L2 session correctly detected `last-wrap-up.txt` had `type: light`. The agent's own thinking block recorded:

> "Last wrap-up=light → §5.9 normally offers /dsm-light-go switch, but user explicitly typed /dsm-go. I'll honor that."

The agent unilaterally pressed past the prompt instead of surfacing it. Auto mode's "minimize interruptions, prefer action over planning" guidance pushed the agent through a step that exists *specifically* to interrupt and ask. The user was never given the choice §5.9 mandates.

Combined effect: layer 2 swallowed the only existing user-facing safety prompt for "light wrap-up means continuation"; layer 1 means there is no second safety net once that prompt is bypassed.

### A third related observation , session-number ambiguity

The S10.L2 session ran with the transcript header still reading "Session 10 Transcript" (the /dsm-go continuation skipped Step 6 reset). The parallel session was apparently started under the same "S10 continuation" framing. There is no canonical name for "the same session resumed in a new conversation window" and no convention for distinguishing it from a fresh session N+1. The user resolved this manually by naming this session `s10.L2` (S10, leg 2). The lack of a naming convention is itself a symptom of the missing concurrent-session protocol , without a notion of "the active session", there is no need to disambiguate active continuations.

## Backlog proposal: BL , concurrent-session detection (hard halt)

### Problem

DSM has no mechanism to detect that another Claude Code conversation is already operating on the same project. Multiple sessions on the same branch produce: interleaved transcript entries, conflicting baseline files, race conditions on staged changes, and confused wrap-up flows.

### Proposed mechanism

Add a session lockfile at `.claude/session.lock` written by `/dsm-go` Step 0.something (e.g., Step 0.7 between branch setup and Step 1) and cleared by every wrap-up skill (full, quick, light, parallel).

**Lockfile contents:**

```
session: <N or N.Lk>
started: <ISO 8601>
agent: <harness identifier>
model: <model identifier>
branch: <git branch>
transcript_anchor: <byte offset or last-line hash of session-transcript.md at start>
pid: <Claude Code process PID, if available>
```

**`/dsm-go` Step 0.7 (new):**

1. If `.claude/session.lock` does not exist → proceed normally; write the lockfile after Step 0 (branch setup).
2. If `.claude/session.lock` exists:
   a. Read its contents.
   b. **Halt with the lockfile contents printed** and the message:
      > "Concurrent session detected. Session {N} started at {timestamp}, agent {agent}, model {model}, branch {branch}. Wrap that session up first (`/dsm-wrap-up` or `/dsm-light-wrap-up`), or pass `--force-concurrent` if you have a reason to override (e.g., the lockfile is stale from a crashed session)."
3. **Hard halt with `--force-concurrent` escape hatch.** Soft warnings get clicked through; the failure mode is the exact one that produced this feedback.

**Wrap-up skills add a new last step:**

- Light, quick, full, parallel wrap-ups: `rm .claude/session.lock` (or `git rm` if tracked, but the file should be in `.claude/` which is gitignored by Claude Code's exclude rule, so it should not be tracked).

**Stale-lockfile recovery:**

- `--force-concurrent` removes the existing lockfile and writes a fresh one.
- Document that crashed-session lockfiles can also be removed manually: `rm .claude/session.lock` then re-run `/dsm-go`.

### Companion BL: extend `/dsm-go` Step 5.9 to bypass auto-mode suppression

Step 5.9's prompt should be marked as **non-suppressible** in auto mode. Add a convention to DSM_0.2 (or a metadata flag in skill specifications) for prompts that auto mode must not skip. Candidate name: "user-mandatory prompts" or "non-suppressible gates". Apply the convention to:

- `/dsm-go` Step 5.9 (light-wrap-up continuation prompt)
- `/dsm-go` Step 0.7 (concurrent-session detection, this BL)
- Any future safety prompt where the user's choice changes the agent's destination skill

The intent is captured by the user's instruction in this session ("auto mode swallowed the prompt I should have been asked"), and the mechanism is a prompt classification that auto-mode honors.

### Companion BL: session naming convention for resumed-in-new-window continuations

Add a §X subsection to DSM_0.2 specifying:

- Original session: `sN`
- Resumed in new window without a prior wrap-up (problematic, should not happen post-this-BL): `sN.Lk` where k is the leg number (2 for the first resumption, 3 for the second, etc.)
- Resumed in new window after a light wrap-up (expected continuation case): `sN.Lk` is acceptable, OR allocate `sN+1` if the user prefers a fresh session number.

This convention gives wrap-up files, feedback files, and MEMORY entries a way to disambiguate the cases described above. The naming convention is a documentation artifact; the lockfile BL is the enforcement mechanism. Both are needed.

### Priority

**High.** The failure produces silent data hazards (uncommitted file collisions, transcript corruption, MEMORY drift) and only surfaces when the user manually reads `git status` and notices unfamiliar entries. Every project with multi-window usage is exposed.

### Origin

Session 10.L2 (2026-04-29). Failure detected when `git status` showed `dsm-docs/decisions/2026-04-24_phase2-embedding-model-selection.md` as untracked despite no creation in this session's command history. Root-cause analysis surfaced both layers documented above.
