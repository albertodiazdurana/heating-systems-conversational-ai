## BL proposal: Resource-Aware Sprint Planning

**Type:** Backlog Proposal
**Priority:** Medium
**Source:** heating-systems-conversational-ai S9

### Problem

Sprint plans currently treat compute as an infinite resource. DSM_0.2 §14
(Module A) sets a session-scoped baseline (model / effort / thinking);
DSM_0.2 §8.7 (BL-402) adjusts per-artifact inside Gate 1. Neither
operates at *sprint* granularity.

When weekly usage limits bite mid-sprint, the spoke has no
protocol-level guidance for reordering work against pool availability.
Without an explicit sprint-level resource view, the default behavior is
ad-hoc: either "burn through the remaining weekly budget and hope," or
"defer everything until reset," both suboptimal.

**Observed incident (S9 open, Tue 2026-04-21 13:00 CEST):**
- all-models weekly: 91% used, 48 h to reset (Thu 21:00)
- sonnet-only: 4% used, plenty of headroom (reset Mon 09:59)
- Sprint 2 Phase 1 Gate 1 (architectural, Opus-needed) was next up per
  the plan, which would have over-consumed the scarce pool

The spoke resolved this locally by adding a "Resource-Aware Execution
Order" section to the Sprint 2 plan, classifying each task by pool
(all-models | sonnet-only | claude-design | extra-usage) and reordering
work so nothing is deferred, only rerouted. This local convention is a
candidate for DSM Central formalization.

### Proposal

Add a new DSM_0.2 subsection (proposed §X.Y "Resource-Aware Sprint
Planning") and update DSM_2.0.C §1 Template 8 to include a "Resource
Budget" and "Resource-Aware Execution Order" section by convention.

Template 8 additions:

1. **Resource Budget header block** (one table, filled at sprint open):
   pool state at sprint start (all-models %, sonnet-only %, claude-design
   %, extra-usage state) + reset cadence anchors + subscription tier.
2. **Resource-Aware Execution Order section** (new `## ` section between
   "Deliverables" and "Phases"): task × pool table with rationale and
   window (pre-reset / post-reset-of-pool-X), plus ordering heuristic
   and re-enter-Gate-1 trigger criteria.

DSM_0.2 §X.Y protocol additions:

1. **Ordering heuristic** (prescriptive): front-load reading-heavy and
   mechanical work into whichever pool has the most headroom. Reserve
   architectural reasoning, language-quality writing, and synthesis
   for the pool that will have fresh capacity at the planned decision
   window. The goal is to avoid deferring work; reorder against pool
   topology instead.
2. **Gate 1 re-entry trigger**: if pool state crosses a threshold that
   invalidates the ordering (e.g., formerly ample pool exceeds 60%
   utilization, or extra-usage is toggled mid-sprint), pause and
   re-brief before continuing.
3. **Granularity asymmetry** (relationship paragraph): §14 is
   session-scoped, §8.7 is artifact-scoped, the new §X.Y is
   sprint-scoped. The three form a nested hierarchy: sprint resource
   plan sets the corridor, session baseline picks a point within it,
   artifact-level recommendations adjust within the session.

### Grounding

- **DSM_6.0 Resource and Procurement Planning**: compute is a
  procurement input. Sprint-level resource planning instantiates the
  procurement discipline at the right scale (sprint = the smallest
  unit of committed work, which is the same scale procurement decisions
  are made in project management).
- **DSM_0.2 §8.7 BL-402**: asymmetric trigger already exists at
  artifact granularity. Extending it to sprint granularity is the
  natural next step; the same discipline (recommendation only when
  demand diverges from baseline) applies.
- **DSM_0.2 §14**: session baseline works below the sprint layer.
  Without a sprint-scoped analog, the baseline must be re-derived each
  session, losing context about multi-session sprint arcs.
- **PMP Procurement Knowledge Area**: identify resource constraints
  before scheduling. Sprint plans currently schedule without surfacing
  the compute constraint.

### Test / acceptance

- Sprint plan template (DSM_2.0.C §1 Template 8) gains the two sections.
- At least one in-flight sprint plan adopts the pattern as a real
  smoke-test before promotion (S9 Sprint 2 local adoption is the
  smoke-test).
- Ordering heuristic produces a non-trivial reorder on at least one
  spoke with >50% utilization on any single pool.

### Origin trace

S9 open: spoke at 91% all-models used, 48h to reset. Agent initially
proposed "defer architectural work to post-reset." User rejected the
deferral frame: "I will try not to defer any work, let's plan in the
order that is best for the project." That reframe, no deferral, only
reordering against pool topology, is the novel contribution. It maps
cleanly onto PMP procurement discipline applied at sprint scale and is
not yet codified anywhere in DSM.

## BL proposal: DSM Provenance DAG as a canonical artifact type

**Type:** Backlog Proposal
**Priority:** Medium
**Source:** heating-systems-conversational-ai S9

### Problem

DSM projects accumulate research files, decision records, experiments, and
external resources that collectively justify why a plan looks the way it does.
These connections exist but are implicit, embedded as prose references scattered
across individual documents. A new reader, or a future session starting cold,
must already know the document graph to navigate it. There is no DSM artifact
type that makes this graph explicit.

**Observed incident (S9):** user asked "why did you do T3, T4, T5 this way?"
The answer required pointing to two separate documents (orchestration decision
+ sprint plan Experiment Gate), neither of which linked to the other directly
from a single navigable entry point. The implicit graph existed but was not
surfaced. A provenance DAG would have answered the question with a single file.

### Proposal

Introduce `dsm_provenance_DAG.md` as a new canonical DSM artifact type with
the following specification:

**Name:** `dsm_provenance_DAG.md`
**Location:** `dsm-docs/plans/` (it exists to support and explain plans;
it is the connective tissue between plans and their justifying artifacts)
**Scope:** project-level living document (spans all sprints, epochs, phases)
**Lifecycle:** generated at project start (empty scaffold), maintained
throughout project implementation, append-only as new artifacts join

**Structure:**
- Plans as backbone nodes (the "spine" of the graph)
- Supporting artifacts as nodes: research, decisions, experiments, external
  resources, inbox/playbooks
- Edges = directed relationships: "feeds", "validates", "frames", "defines",
  "prerequisite", "confirms"
- Format: Mermaid diagram (visual overview) + node registry table (full paths
  and descriptions) + edge registry table (relationship semantics). Table is
  the append target; Mermaid is regenerated when the graph grows materially.

**What it is NOT:**
- Not a summary of the plan (the plan is authoritative)
- Not a decision record (decisions remain in dsm-docs/decisions/)
- Not research (research files remain in dsm-docs/research/)
- A navigation and provenance artifact only; the underlying documents are
  the source of truth

**DSM_2.0.C §1 Template 8 addition:** sprint plans gain a short "Provenance"
header block pointing to `dsm_provenance_DAG.md` and listing which nodes in
the DAG directly scope or gate the sprint. This replaces the current pattern
of burying references in prose prerequisites.

**Naming rationale:** "DAG" reflects the actual structure (directed acyclic
graph, not a linear chain). "Provenance" reflects the purpose (tracing origin
and justification of design choices). "DSM" prefix signals it is a DSM
artifact type, not a project-specific file.

### Grounding

- **DSM_6.0 §1.3 Earn Your Assertions:** the provenance DAG is the artifact
  that makes assertions in a plan traceable to evidence. A plan that references
  "we chose Haystack" without a provenance link to the decision record that
  chose Haystack is an unearned assertion.
- **DSM_6.0 §1.5 Know Your Context:** a future session starting cold should
  be able to read the DAG and understand the full context of any plan element
  without reading all supporting documents.
- **PMP Traceability Matrix:** the DAG instantiates requirements traceability
  at the artifact level, a standard project management practice with no
  current DSM equivalent.
- **DSM_0.2 §7 Session Transcript Protocol:** the transcript records reasoning
  per session; the provenance DAG records the cumulative artifact graph across
  sessions. They are complementary: transcript = session-scoped reasoning log,
  provenance DAG = project-scoped artifact lineage.

### Test / acceptance

- `dsm_provenance_DAG.md` added to the canonical scaffold list in DSM_0.2
  and `/dsm-align` Step 3 table.
- `/dsm-go` Step 3.5 (checkpoint check) or a new Step 3.7 reminds the agent
  to append new artifacts to the DAG at session start if new files were
  created in the previous session.
- At least one spoke project (this one) has a complete DAG before the artifact
  type is promoted to canonical (smoke-test: S9 Sprint 2 DAG is the prototype).

### Origin trace

S9: user observed that answering "why did you do this?" required cross-referencing
two documents. User's insight: "I see the opportunity to design a missing document
that generalizes the connection and relationship between documents to support and
explain a plan." User named the artifact and identified the DAG structure
(correcting the initial "chain" framing to reflect non-linear interdependencies).
The "plans as backbone, supporting artifacts as nodes" framing emerged from the
collaborative design conversation.
